# Undo/Redo UI — Design Spec

**Date:** 2026-07-10
**Status:** Approved design (brainstorm complete)
**Scope:** Pure UI wiring on top of `store.js`'s existing `Project.undo()`/`Project.redo()`, plus one new pure-function addition (`describeChange`) for descriptive tooltips. Touches `src/js/store.js` (new function only, existing undo/redo logic untouched), `src/js/ui/app.js`, `src/index.html`.

## 1. Purpose

`store.js`'s `Project` class has had working `undo()`/`redo()` methods (50-step history, snapshot-based, redo-stack-cleared-on-new-action) since the foundation phase — but no button or keyboard shortcut anywhere in the app has ever called them. This is a known, previously-flagged gap (noted in multiple past final-review passes as "pre-existing, out of scope for this plan"). This spec closes it.

## 2. Decisions Log

| Question | Decision |
|---|---|
| Trigger mechanism | Both: header buttons (`Undo`/`Redo`, disabled when their respective stack is empty) and a keyboard shortcut (`Ctrl+Z`/`Cmd+Z` for undo, `Ctrl+Shift+Z`/`Cmd+Shift+Z` for redo). |
| Shortcut scope | Ignored while `document.activeElement` is a text-editable element (`input`, `textarea`, `select`) — lets the browser's native in-field undo work normally while editing a Plan-tree cell or any other text input, instead of colliding with app-level undo. |
| Button placement | `#header-top`, immediately before the existing `#save-button`. |
| Feedback | Each button's `title` attribute shows a human-readable description of what it would do, generated by a new `describeChange(before, after)` function — e.g. `"Undo: Change plannedStart on 'Design'"`. Not a toast/alert — a hover tooltip, consistent with this app's existing quiet-by-default interaction style (no new dialogs). |
| Persistence | None — undo/redo history is session-only (in-memory `_undoStack`/`_redoStack`), lost on page reload. This matches how virtually every app already behaves (closing/reopening a document clears undo history) and is not a gap to fix — intentionally out of scope. |
| Revision-number interaction | No special handling needed. `undo()`/`redo()` already restore `meta` (including `revision`) as part of the snapshot; the subsequent `refresh(state, true)` bumps it forward by one from whatever value was restored. Revision numbers may not stay strictly monotonic across an undo/redo branch, but since `revision` is only ever used for save/export filenames and the load-time conflict-detection comparison (not for strict ordering guarantees), this has no correctness impact and needs no fix. |

## 3. `describeChange(before, after)` (`store.js`)

A new pure function, exported alongside `Project`. Both `before` and `after` are full project snapshots shaped like `Project.prototype.toJSON()`'s output (`{ meta, tasks, holidays, picList, snapshots, auditLog, settings }`).

Diff logic, in priority order:
1. **Tasks added** (present in `after.tasks`, absent from `before.tasks` by `id`): if exactly one, `"Add '<name>'"`; if more than one, `"Add <N> tasks"`.
2. **Tasks removed** (present in `before.tasks`, absent from `after.tasks`): if exactly one, `"Delete '<name>'"`; if more than one, `"Delete <N> tasks"`.
3. **Tasks with changed fields** (same `id` present in both, but one or more own-properties differ by value, compared via `JSON.stringify` equality): collect the set of changed tasks and, per task, the list of changed field names.
   - Exactly one task changed, with exactly one field changed: `"Change <field> on '<name>'"` (using the task's `after`-state name as the label, even if `name` itself is the field that changed).
   - Exactly one task changed, with more than one field changed: `"Change <N> fields on '<name>'"`.
   - More than one task changed: `"Change <N> tasks"` (covers cascading successor shifts from a drag, and multi-task CSV import).
4. If no task-level difference is found, fall through to whichever of `holidays`, `picList`, `snapshots`, `settings` differs (checked in that order, `JSON.stringify` equality): `"Change holidays"`, `"Change PIC list"`, `"Take snapshot"`, `"Change settings"` respectively.
5. Fallback if nothing above matches (should not occur in practice, but keeps the function total): `"Change"`.

This is a pure function — no DOM, no `Project` instance state — fully Node-testable like every other function in `store.js`.

## 4. UI Wiring (`app.js`, `index.html`)

- `index.html`: two new buttons in `#header-top`, before `#save-button`: `<button id="undo-button">Undo</button><button id="redo-button">Redo</button>`.
- `app.js`'s `refresh(state, markDirty)` gains a call to a new `updateUndoRedoButtons(state)` that runs on every refresh:
  - `undo-button.disabled = state.project._undoStack.length === 0`; if not disabled, `title = 'Undo: ' + describeChange(state.project._undoStack[state.project._undoStack.length - 1], state.project.toJSON())`; if disabled, `title = ''`.
  - `redo-button.disabled = state.project._redoStack.length === 0`; if not disabled, `title = 'Redo: ' + describeChange(state.project.toJSON(), state.project._redoStack[state.project._redoStack.length - 1])`; if disabled, `title = ''`.
- Click handlers: `undo-button` click calls `state.project.undo()`; if it returned `true`, calls `refresh(state, true)`. Same pattern for `redo-button` calling `state.project.redo()`.
- A new `wireUndoRedoKeyboard(state)` function, called once from `showApp(state)`, adds a single `document.addEventListener('keydown', ...)` handler:
  - If `document.activeElement` is an `INPUT`, `TEXTAREA`, or `SELECT`, return immediately (do nothing — let the browser's native field-level undo run).
  - Else if `(e.ctrlKey || e.metaKey) && e.key.toLowerCase() === 'z' && !e.shiftKey`: `e.preventDefault()`, call `state.project.undo()`, `refresh(state, true)` if it returned `true`.
  - Else if `(e.ctrlKey || e.metaKey) && e.key.toLowerCase() === 'z' && e.shiftKey`: `e.preventDefault()`, call `state.project.redo()`, `refresh(state, true)` if it returned `true`.

## 5. Testing

- `store.js`: `describeChange` gets full Node test coverage in `tests/store.test.js` — one test per branch of the priority-ordered diff logic above (single add, multi add, single delete, multi delete, single-field change, multi-field change on one task, multi-task change, holidays/picList/snapshots/settings fallback, and the no-op fallback).
- `app.js` is a UI file with no automated coverage by this project's standing convention (no jsdom) — verified via controller-run Playwright checks: buttons correctly enable/disable across a sequence of edits, clicking each restores/reapplies state and matches the expected `describeChange` tooltip text, keyboard shortcut works when focus is elsewhere, and is correctly ignored (native field-undo behavior preserved) while a Plan-tree cell editor is focused.
- Regression: existing test count must grow by exactly the new `describeChange` tests, with all prior tests still passing.

## 6. Out of Scope

- No change to `store.js`'s existing `undo()`/`redo()`/`_pushUndo()`/`_applyState()` logic — this spec only adds one new pure function alongside them.
- No persistence of undo/redo history across page reloads (see Decisions Log).
- No fix to the pre-existing gap where `addTask`/`deleteTask`/`moveTask` don't write audit log entries (noted in an earlier final-branch review as a separate, unrelated pre-existing gap) — `describeChange` derives its description from direct state-diffing, not from the audit log, so it works correctly regardless of that gap.
